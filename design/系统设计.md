# SSS-Api 系统设计

## 1. 设计目标

`SSS-Api` 的目标不是再做一个单一平台的中转脚本，而是做一个面向多平台、多租户、多账号池的 AI 中转站。

结合仓库中的三个参考项目，建议把新系统拆成三层能力：

1. 产品骨架层：参考 `claude-relay-service`
2. 平台能力层：参考 `sub2api`
3. 协议适配层：参考 `CLIProxyAPI`

最终目标技术栈保持当前项目约束：

- 后端：Python + `uv` + FastAPI
- 前端：Vue 3 + Vite

本设计文档只抽象能力边界，不直接照搬 Node.js / Go 的实现细节。

## 2. 总体结论

### 2.1 为什么以 `claude-relay-service` 为主体骨架

`claude-relay-service` 最接近“我自己搭一个能用、能管、能拼车、能看统计的 AI 中转服务”这种产品心智。

直接证据：

- 主入口和整体路由组织在 `claude-relay-service/src/app.js`
- Claude 主链路在 `claude-relay-service/src/routes/api.js`
- OpenAI 兼容统一入口在 `claude-relay-service/src/routes/unified.js`
- OpenAI Responses / Codex 入口在 `claude-relay-service/src/routes/openaiRoutes.js`
- 它已经把“管理后台 + 用户 + API Key + 账户池 + 使用统计”串成了完整产品流程

从 README 也能看出它的产品定位是：

- 多账户管理
- 自定义 API Key
- 使用统计
- 监控面板
- 安全控制
- 代理支持

这些能力都直接体现在 `claude-relay-service/README.md` 和 `claude-relay-service/src/app.js` 的路由、静态后台、健康检查、指标接口中。

结论：

- 新项目的后台操作流、账号管理体验、API Key 使用方式，可以优先继承它的产品形态
- 但不能继承它“Redis 承担过多业务真相 + 偏单站点拼车”的架构习惯

### 2.2 为什么以 `sub2api` 为多租户与平台能力参考

`sub2api` 的价值不在“某一条转发链路更简单”，而在它已经把平台能力建完整了。

直接证据：

- 管理端路由结构在 `sub2api/backend/internal/server/routes/admin.go`
- 用户侧路由在 `sub2api/backend/internal/server/routes/user.go`
- 网关总入口在 `sub2api/backend/internal/server/routes/gateway.go`
- 管理端页面结构在 `sub2api/frontend/src/router/index.ts`
- 领域模型在 `sub2api/backend/ent/schema`

它的核心不是单独一个 `tenant` 字段，而是一整套组合模型：

- 用户：`sub2api/backend/ent/schema/user.go`
- 分组：`sub2api/backend/ent/schema/group.go`
- 账号：`sub2api/backend/ent/schema/account.go`
- API Key：`sub2api/backend/ent/schema/api_key.go`
- 用户订阅：`sub2api/backend/ent/schema/user_subscription.go`
- 兑换码：`sub2api/backend/ent/schema/redeem_code.go`
- 使用日志：`sub2api/backend/ent/schema/usage_log.go`
- 套餐：`sub2api/backend/ent/schema/subscription_plan.go`
- 渠道与模型定价：`sub2api/backend/migrations/081_create_channels.sql`

结论：

- 新项目的“多租户、计费、订阅、CDK、统计、渠道定价”应该主要借鉴 `sub2api`
- 但不能把它当前全部业务面一次性照搬，否则第一阶段范围会过大

### 2.3 为什么以 `CLIProxyAPI` 为协议转换参考

`CLIProxyAPI` 最强的地方不是后台，而是把“协议转换”和“上游执行器”严格拆开了。

直接证据：

- 服务端基础入口在 `CLIProxyAPI/internal/api/server.go`
- 协议转换注册表包装在 `CLIProxyAPI/internal/translator/translator/translator.go`
- 翻译器目录在 `CLIProxyAPI/internal/translator`
- 执行器目录在 `CLIProxyAPI/internal/runtime/executor`
- 设计说明在 `CLIProxyAPI/docs/sdk-advanced_CN.md`

它明确区分：

- translator：负责协议之间的 request / response 转换
- executor：负责真正打到上游的 HTTP / SSE / WebSocket 调用

结论：

- 新项目必须继承这种分层思想
- 但不应继承它“文件配置 + 管理 API + auth 文件中心化”的全部产品形态；这里只借协议适配方法，不借整套管理模型

## 3. 三个参考项目分别借什么、不借什么

| 项目 | 重点借鉴 | 不建议直接继承 |
|------|----------|----------------|
| `claude-relay-service` | 产品骨架、后台操作流、API Key 体验、账号池调度感受、统一入口思路 | Redis 作为过多业务真相、偏单站点拼车的数据模型、路由与业务强耦合 |
| `sub2api` | 分组/账号/API Key/订阅/兑换码/支付/统计/渠道定价/运维后台 | 业务面过宽、概念过多，第一阶段不要一次搬全 |
| `CLIProxyAPI` | translator/executor 分层、协议矩阵、provider-specific path、管理接口抽象 | 文件配置中心、偏 CLI 本地代理的产品形态、只做统计不做完整业务计费 |

## 4. 推荐的目标架构

## 4.1 架构分层

建议把 `SSS-Api` 分成 6 个核心层次。

### A. Client Gateway 层

职责：

- 暴露统一客户端入口
- 负责鉴权、限流、请求标准化、协议识别
- 对外提供：
  - `/v1/messages`
  - `/v1/messages/count_tokens`
  - `/v1/models`
  - `/v1/chat/completions`
  - `/v1/responses`
  - `/v1/responses/compact`
  - `/v1beta/models/*`

参考：

- `sub2api/backend/internal/server/routes/gateway.go`
- `claude-relay-service/src/app.js`
- `CLIProxyAPI/internal/api/server.go`

### B. Protocol Adapter 层

职责：

- 把不同客户端协议转换成内部规范请求
- 再把内部请求转换成目标上游协议
- 反向把上游响应转换回客户端原始协议

建议按两级拆分：

1. `inbound adapter`
   - `anthropic_messages`
   - `openai_chat`
   - `openai_responses`
   - `gemini_native`
2. `provider adapter`
   - `anthropic`
   - `openai`
   - `codex`
   - `gemini`
   - `antigravity`
   - `custom_openai_compatible`

参考：

- `CLIProxyAPI/internal/translator`
- `CLIProxyAPI/docs/sdk-advanced_CN.md`
- `sub2api/backend/internal/pkg/apicompat`
- `sub2api/backend/internal/service/openai_gateway_messages.go`

### C. Scheduler / Account Pool 层

职责：

- 多账号选择
- 粘性会话
- 故障转移
- 限流恢复
- 账号可调度状态判断
- 账号隐私配置合规校验

这里建议吸收两边优点：

- 调度字段参考 `sub2api/backend/ent/schema/account.go`
- 调度体验参考 `claude-relay-service` 的 unified schedulers

关键能力：

- `session_hash`
- `group -> accounts`
- `priority`
- `schedulable`
- `rate_limited_at`
- `rate_limit_reset_at`
- `overload_until`
- `temp_unschedulable_until`
- `privacy_mode`

### D. Platform Business 层

职责：

- 多租户
- 用户与成员关系
- API Key 生命周期
- 订阅、余额、配额、CDK、渠道定价
- 临时号池分配
- 支付与订单

这个层主要参考 `sub2api`，但要补一个 `tenant` 顶层实体，因为当前参考项目里多是“站点级平台”模型，不是显式 SaaS 多租户模型。

### E. Usage / Billing / Audit 层

职责：

- token 记录
- 成本换算
- 账单归因
- 模型映射链路记录
- 入站/上游端点记录
- 日志脱敏
- 请求详情审计

参考：

- `sub2api/backend/ent/schema/usage_log.go`
- `claude-relay-service/src/services/requestDetailService.js`
- `CLIProxyAPI/internal/usage/logger_plugin.go`

### F. Console / Admin 层

职责：

- 平台管理员后台
- 租户管理员后台
- 用户自助端
- 运维大盘

页面信息架构建议融合：

- 产品操作流参考 `claude-relay-service`
- 页面颗粒度参考 `sub2api/frontend/src/router/index.ts`

## 4.2 推荐的部署视图

建议初期做“单体应用 + 内部分层 + 异步任务”的结构，不急着微服务。

推荐部署结构：

- `FastAPI app`
  - HTTP API
  - Admin API
  - Gateway API
- `Worker`
  - usage 异步入库
  - token 刷新
  - 限流恢复
  - 账号健康检查
  - 订阅过期处理
  - 临时号池回收
- `PostgreSQL`
  - 业务真相数据库
- `Redis`
  - 并发计数
  - 粘性会话
  - 短期 usage cache
  - OAuth session
  - 队列 / 锁 / 限流状态

结论：

- 业务真相必须放 PostgreSQL，不要沿用 `claude-relay-service` 的 Redis 主存思路
- Redis 只做运行时状态和缓存，采用 `sub2api` 的思路更稳

## 5. 推荐的数据模型

## 5.1 顶层模型

建议在 `sub2api` 原有模型之上增加显式租户边界：

```text
Tenant
  ├─ TenantMember
  ├─ Workspace / Group
  │    ├─ APIKey
  │    ├─ ProviderAccountBinding
  │    ├─ UserSubscription
  │    └─ ChannelBinding
  ├─ SubscriptionPlan
  ├─ RedeemCode
  ├─ TempResourcePool
  ├─ PaymentOrder
  └─ UsageLog
```

## 5.2 推荐实体

### 1. `tenant`

新增顶层实体，解决真正的多租户隔离问题。

核心字段建议：

- `id`
- `name`
- `status`
- `owner_user_id`
- `plan_type`
- `settings_json`

### 2. `workspace_group`

保留 `sub2api` 的 `group` 思想，但明确它属于某个租户。

参考：

- `sub2api/backend/ent/schema/group.go`

建议字段：

- `tenant_id`
- `name`
- `platform`
- `subscription_type`
- `rate_multiplier`
- `default_validity_days`
- `claude_code_only`
- `allow_messages_dispatch`
- `require_oauth_only`
- `require_privacy_set`
- `default_mapped_model`

### 3. `provider_account`

保留 `sub2api` 的账户模型，并吸收 `claude-relay-service` 的多类型账户经验。

参考：

- `sub2api/backend/ent/schema/account.go`
- `claude-relay-service/src/services/account/*`

建议支持的账号类型：

- `oauth`
- `setup-token`
- `api-key`
- `cookie`
- `upstream-openai-compatible`

### 4. `api_key`

保持“用户拿 API Key 使用平台”的核心模式。

参考：

- `sub2api/backend/ent/schema/api_key.go`
- `claude-relay-service/src/services/apiKeyService.js`

建议字段：

- `tenant_id`
- `user_id`
- `group_id`
- `key`
- `quota`
- `quota_used`
- `rate_limit_5h`
- `rate_limit_1d`
- `rate_limit_7d`
- `ip_whitelist`
- `expires_at`

### 5. `subscription_plan` 与 `user_subscription`

直接沿用 `sub2api` 的拆法，而不是把订阅信息混进 API Key。

参考：

- `sub2api/backend/ent/schema/subscription_plan.go`
- `sub2api/backend/ent/schema/user_subscription.go`

### 6. `redeem_code`

用于 CDK / 卡密兑换，不要和优惠码混成一个模型。

参考：

- `sub2api/backend/ent/schema/redeem_code.go`
- `sub2api/backend/internal/server/routes/admin.go`

### 7. `channel_pricing`

保留 `sub2api` 的渠道模型定价能力，但可先做简化版。

参考：

- `sub2api/backend/migrations/081_create_channels.sql`

用途：

- 同一个模型对不同租户 / 渠道可以有不同售价
- 适合后续做分销、代理商、企业客户

### 8. `temp_resource_pool`

这个能力三套参考项目都没有完整现成实现，建议独立建模，不要硬塞进 `provider_account`。

建议拆成：

- `temp_resources`
- `temp_resource_leases`
- `temp_resource_events`

适用对象：

- 临时号
- 临时邮箱
- 一次性验证码辅助资源

核心原则：

- 和 AI 上游账号池分开
- 它属于“辅助登录授权资源池”，不是“推理账号池”

### 9. `usage_log`

直接参考 `sub2api` 的字段深度。

参考：

- `sub2api/backend/ent/schema/usage_log.go`

建议至少记录：

- `tenant_id`
- `user_id`
- `api_key_id`
- `account_id`
- `group_id`
- `subscription_id`
- `channel_id`
- `request_id`
- `inbound_endpoint`
- `upstream_endpoint`
- `requested_model`
- `billing_model`
- `upstream_model`
- `input_tokens`
- `output_tokens`
- `reasoning_tokens`
- `cache_creation_tokens`
- `cache_read_tokens`
- `image_count`
- `total_cost`
- `actual_cost`
- `rate_multiplier`
- `account_rate_multiplier`
- `stream`
- `duration_ms`
- `first_token_ms`
- `status_code`

## 8. 第一阶段最小可用版本

建议第一阶段只做必须闭环，不追求一次性覆盖全部功能。

## 8.1 必须实现

- OpenAI / Claude / Gemini / Codex 四类入口中的核心三类：
  - `/v1/messages`
  - `/v1/chat/completions`
  - `/v1/responses`
- 账号池管理
- OpenAI / Claude / Gemini OAuth 辅助登录
- 用户 / 分组 / API Key
- 使用日志与 token 计费
- 基础后台和用户端
- 基础兑换码能力
- 基础数据脱敏

## 8.2 第二阶段再做

- 渠道定价
- 支付与充值
- 完整订阅售卖
- 临时号池分配
- 运维高级面板
- 更复杂的 provider-specific path
- 更多第三方 OpenAI 兼容上游

## 9. 最终建议

### 9.1 应该保留的核心设计

- 产品骨架走 `claude-relay-service`
- 数据与平台模型走 `sub2api`
- 协议转换分层走 `CLIProxyAPI`

### 9.2 不该继续沿用的做法

- 不要把业务真相继续堆在 Redis
- 不要把每个平台逻辑都写进单个大路由文件
- 不要把协议转换和上游调用混在一个 handler 里
- 不要在第一阶段把 `sub2api` 全部后台功能照抄过来

### 9.3 一句话架构结论

`SSS-Api` 应该做成“以 `claude-relay-service` 为产品外壳、以 `sub2api` 为业务中台、以 `CLIProxyAPI` 为协议适配内核”的 FastAPI 平台，而不是三者的简单拼接版。
